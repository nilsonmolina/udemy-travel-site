Section 2: Git Essentials

2 - WHAT IS GIT?

As the title describes, we are starting off this training by learning about Git. We begin by answering the following three questions:

  1. What is Git?

    Git is the world's most popular version control system. It helps us manage our project files by providing History, Collaboration, and Feature Branches. History just means that Git keeps track of every change we make to our projects. Using Git, we can see when the changes were made and on which files, plus it will even show us which lines were added and which lines were removed. Git also helps with collaboration, because it allows multiple people to work on a copy of the latest version of the code at the same time and it will even handle the merging of any conflicts. Lastly, feature branches allows you to work on multiple tasks at the same time, but only commit the completed tasks to the repository without worrying about whether both tasks were completed or not. For example, lets say you were working on redisigning a header and footer. Without Git, if you completed the header, but the footer was still incomplete, you could not upload your changes because you would also be uploading your incomplete footer code.

  2. Why should we learn Git

    It is essentially an industry standard these days and in the developer world, it is sort of assumed that you know Git. So whether you want to seem more attractive to employers or you want to bring structure to your code, then learning Git is truly a must.

  3. Where do we begin?

    Good news, we've already begun. We know what Git is and we know why we should learn it. The next step is to see it in action and thats just what we are going to do in the following module. So lets get our feet wet!

4 - GIT YOUR HANDS DIRTY!

To begin, we will need to download and install Git from the homepage git-scm.com. Installion is a very basic process and you can simply accept all the default values along the way. Git may already be installed on macOS machines and you can check by running the following command in terminal: 

git --version

Now that Git is installed, let's start by personalizing our settings. When Git tracks your projects, it also tracks who is making those changes. So we want to let Git know about your name and your email by running the following commands. 

git config --global user.name "Nilson"
git config --global user.email "molinanilson@gmail.com"

Git is all set up and ready to go. Now let's get Git to actually do something! First let's create a new folder and with our CLI let us navigate to it. The following is what I used to navigate to my newly created 'hello-world' folder: 

cd /c/Users/W10-Dev/Desktop/files/dev/projects/hello-world

With our CLI currently pointing to the root of our new folder, we can now create a repository for our project. 

git init

We have now created our first repo! To test it out lets create a new file using the command below. This will create an empty html file that we can use to create a very simple webpage where we can simply write "hello world". 

touch "index.html"

With this new file created, we have now made a change to our local repo. To see what changes have been made, we can run the following command: 

git status

The "git status" command returns a list of all the changes performed on the repo. Now we want to add our new file to the staging area so that we can then commit this file to our repository. 

git add index.html

Now our file is added to the staging area, we can run the "git status" command again to confirm our staging area, and when we are ready to commit, we can run the following command. 

git commit -m 'My first commit'

Awesome! What makes this so cool is that if we ever make changes to this code, we can always revert back to it. Whether the files are edited or even if the files are deleted, we can simply revert back to this state by using the following command: 

git checkout -- .

Up to this point, we have been working on our local repository. But as our final step, we will instead clone an existing repo from a remote server to our hard drive. Let us pretend that a coworker is working on a project and they want us to help them. So the repo already exists, so we wouldn't create a new repo. Instead we would want to take the existing repo that exists in a server and clone it to our computer. So back in our machine, we will navigate to our projects folder (NOT our 'hello world' repo folder) and then clone the repo to this location. Below are the commands required to do these two steps: 

cd ..
git clone https://github.com/LearnWebCode/welcome-to-git.git

And thats it! If you check the projects folder, you will see the new folder we just cloned.

Section 3: Intro to Node.js & NPM

6 - THE NEED FOR AUTOMATION & ORGANIZATION

While we build our website, we want automation and organization. Automation will save us time and organization will save our sanity.

  1. Automation

    Automation just means to take a task and find a way to have our computer do it for us. So if there is a small task that we find ourselves performing again and again and again, we can save a lot of time if we can make the computer perform that task for us. A good example is automatic browser refresh. Everytime we make a change to our style sheet or html file, instead of hopping into our browser and refreshing the page everytime, we can have our computer detect that changes were saved to our file and then automatically trigger a refresh in the browser for us.

  2. Organization

    Organization is crucial and will not only save us time when we need to find a certain file, but it will also help us when we come back to our code, months or even years later. In this course we will touch upon CSS Organization and Package Management. With CSS Organization, we can create multiple smaller and maintainable css files for us to work with. The downside to this is that browsers will now have to download all of these multiple files. Fortunately, node allows us to combine all of those separate files into a single stylesheet for the web browser, giving us the best of both worlds. Package Management on the other hand allows us to download all of our libraries (ie jquery, bootstrap, normalize.css, etc...) without having to use a browser and go to all of their respective websites. Package Management has a lot of benefits that we will see first hand in the modules to come.

7 - NODE.JS INTRODUCTION

This lesson has three main parts to it:

  1. Figure out what node.js is
  2. Install node on our computer
  3. Perform a hands on demo

Traditionally, JavaScript is a language that only runs in the web browser. Node.js changes this, and allows javascript to run on the server. It essentially takes out the javascript guts from google Chrome and creates a new playground located on the computer for javascript to run on.

To install node to your local machine, simply open up your web browser of choice and go to the node home page: http://nodejs.org/en/ and download the LTS version. We don't want the latest and most cutting edge version, and will instead get the latest stable version. Once downloaded, simply double click it and install it with all the default values. After installation is completed, open up the CLI and then check to see your version number by using the following command: 

node -v

Now lets get a brief hands on with running javascript without a browser. Create a new folder, I called mine sandbox, and within that folder create a new javascript file, I named mine test.js. Open up the javascript file and type in console.log(2+2); save the file and then navigate to the folder using a CLI. In the CLI, run the javascript file using the following command: 

node test.js

You should get a response of "4" in the CLI. This may not seem like a big deal, but the significance of this is that the contents of the Javascript file were not merely repeated or simply echoed out as text. Instead, node actually interpreted and evaluated our javascript code! Node calculated "2 + 2" and responded with 4.

Lastly, we played with a few extra node capabilities by allowing javascript to create files on the file system, and even download a picture from the internet. The code can be found in the github repository under: udemy-travel-site/sandbox/3-7-node-experiment folder.

8 - NPM INTRODUCTION

In this lesson, we are going to learn about NPM or Node Package Manager. We will be covering three main things:

  1. What is NPM? What is package management? Why would we use it?

    Think of NPM as a Grocery Store that contains code instead of food. If we were going to bake a cake, we would need flour, sugar, baking powder, etc. How would we get those ingredients? For example, with flour, would we start with wheat and turn that into flour ourselves? Maybe some people would, but the grand majority would go to the grocery store and pick up some flour. The same is true with development of code. When baking a... well a website, the ingredients may be jQuery, picturefill, auto compile css, etc. Instead of writing most of these things ourselves, we can go to the "grocery store", NPM, and pickup what we need. So NPM is a centralized place where developers share their code with the world for free. Without NPM we would need to go to multiple "grocery stores" or websites, to get all of our ingredients. This is the equivalent of going to bootstraps website for bootstrap, jquery's website for jquery, and so on and so forth. With package management, we can simply make a list of packages we want, and we will get all of those automatically. To continue with our metaphor, what happens when our ingredients get old or expire? We go back to the grocery store and replace our old ingredients with the fresh and new one. Same is true with package management, and it can be done with a single command! So not only can we easily get all of the packages (ingredients) that we need, but we can also get any updates on those packages as needed, very easily.

  2. What types of files/packages we hope to find on NPM

    Most packages will fall in one of two categories: Node packages,(typically enable us to perform automation, build tools, server tasks) & Other things (like jQuery, Lodash, Bootstrap, Normalize.css, etc.)

  3. Hands-on demo

    First lets add jquery to our project. To do so, open up your CLI and navigate to the project folder. After that, run the following command to add jquery to this project: 

    npm install jquery

    This creates a new directory (folder) called "npm_modules" and within that is the jquery library that we requested. So that did exactly what we wanted, but this still requires us to download packages one by one. Instead, lets delete the node_modules folder and try a more organized approach to getting our packages. Lets create our "grocery list" or "recipe ingredients list" of all the packages we will need for our project. To do that lets run the following command: 

    npm init

    The CLI will ask us for more information, for now let's leave it as is and accept the defaults by hitting the "enter" key until it is done. This will create a new file called "package.init" which is our "grocery list" and we will want to create one of these for every one of our projects. We can open the file with our text editor of choice and we will see that it currently only contains information on our project. For now, we can simply delete everything except name and version. What makes the package.json file so great, is that we can use it to keep track of which packages we are using. Now if we use the command below, we will see that the package.json file has updated. 

    npm install jquery --save
    npm install normalize.css --save

    If you open up the package.json file now, you will see that a dependencies collection was added to the file that includes the jquery and normalize.css packages that we added.

    One example of why this is very useful can be seen be deleting the node_modules folder that contains the packages that we just installed. By doing this, we lost the packages, but because we have our "recipe list" we can simply retrieve those packages by running the following command: 

    npm install

    When we run "npm install", node looks in the directory that our CLI is currently pointed at and it looks for the package.json file. Then it will install all the dependencies in the package.json file in one fell swoop.

    To figure out the exact name of packages, you can go to the npm homepage and use the search functionality. Or you can simply google and use "npm" at the end of your search query.

    And lastly, if you run "git status" command, you will notice that the node_modules folder does not show up. Even if you push your local repository to github, you will notice that the node_modules are NOT uploaded. This is because in our ".gitignore" file, we are telling git to ignore the node_modules folder. The gitignore file can help us keep our repositories clean and is especially useful for files such as .DS_Store which is found in macOS and other such things. We tell git to ignore the node_modules folder because it is a best practice. As long as we have the package.json file, we can simply run "npm install" to get all those files again.
