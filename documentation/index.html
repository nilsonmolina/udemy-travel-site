Section 2: Git Essentials

2 - WHAT IS GIT?

As the title describes, we are starting off this training by learning about Git. We begin by answering the following three questions:

  1. What is Git?

    Git is the world's most popular version control system. It helps us manage our project files by providing History, Collaboration, and Feature Branches. History just means that Git keeps track of every change we make to our projects. Using Git, we can see when the changes were made and on which files, plus it will even show us which lines were added and which lines were removed. Git also helps with collaboration, because it allows multiple people to work on a copy of the latest version of the code at the same time and it will even handle the merging of any conflicts. Lastly, feature branches allows you to work on multiple tasks at the same time, but only commit the completed tasks to the repository without worrying about whether both tasks were completed or not. For example, lets say you were working on redisigning a header and footer. Without Git, if you completed the header, but the footer was still incomplete, you could not upload your changes because you would also be uploading your incomplete footer code.

  2. Why should we learn Git

    It is essentially an industry standard these days and in the developer world, it is sort of assumed that you know Git. So whether you want to seem more attractive to employers or you want to bring structure to your code, then learning Git is truly a must.

  3. Where do we begin?

    Good news, we've already begun. We know what Git is and we know why we should learn it. The next step is to see it in action and thats just what we are going to do in the following module. So lets get our feet wet!

4 - GIT YOUR HANDS DIRTY!

To begin, we will need to download and install Git from the homepage git-scm.com. Installion is a very basic process and you can simply accept all the default values along the way. Git may already be installed on macOS machines and you can check by running the following command in terminal: 

git --version

Now that Git is installed, let's start by personalizing our settings. When Git tracks your projects, it also tracks who is making those changes. So we want to let Git know about your name and your email by running the following commands. 

git config --global user.name "Nilson"
git config --global user.email "molinanilson@gmail.com"

Git is all set up and ready to go. Now let's get Git to actually do something! First let's create a new folder and with our CLI let us navigate to it. The following is what I used to navigate to my newly created 'hello-world' folder: 

cd /c/Users/W10-Dev/Desktop/files/dev/projects/hello-world

With our CLI currently pointing to the root of our new folder, we can now create a repository for our project. 

git init

We have now created our first repo! To test it out lets create a new file using the command below. This will create an empty html file that we can use to create a very simple webpage where we can simply write "hello world". 

touch "index.html"

With this new file created, we have now made a change to our local repo. To see what changes have been made, we can run the following command: 

git status

The "git status" command returns a list of all the changes performed on the repo. Now we want to add our new file to the staging area so that we can then commit this file to our repository. 

git add index.html

Now our file is added to the staging area, we can run the "git status" command again to confirm our staging area, and when we are ready to commit, we can run the following command. 

git commit -m 'My first commit'

Awesome! What makes this so cool is that if we ever make changes to this code, we can always revert back to it. Whether the files are edited or even if the files are deleted, we can simply revert back to this state by using the following command: 

git checkout -- .

Up to this point, we have been working on our local repository. But as our final step, we will instead clone an existing repo from a remote server to our hard drive. Let us pretend that a coworker is working on a project and they want us to help them. So the repo already exists, so we wouldn't create a new repo. Instead we would want to take the existing repo that exists in a server and clone it to our computer. So back in our machine, we will navigate to our projects folder (NOT our 'hello world' repo folder) and then clone the repo to this location. Below are the commands required to do these two steps: 

cd ..
git clone https://github.com/LearnWebCode/welcome-to-git.git

And thats it! If you check the projects folder, you will see the new folder we just cloned.

5 - SETUP YOUR GITHUB

In this lesson we have two main tasks

  1. Create a GitHub account
  2. Create the repo with our website's starter files

GitHub allows us to store our repository to a remote server. I am going to skip ahead a bit, but essentially we need to create a GitHub account and then create a new repository. At this point, we now have a place to remotely store our repositories. This protects us in the event where our local computer crashes, gets stolen, explodes... you get the point. It also allows us to share our repo and if we choose, we can allow others to make changes and bugfixes to the code. Now that we have a new and completely empty repo, it is waiting for us to push our local repo to the remote repo in GitHubs servers. 

cd /c/Users/W-10-Dev/Desktop/files/dev/projects
git clone https://github.com/LearnWebCode/travel-site-files.git
cd udemy-travel-site/
git status
git add .
git commit -m 'cloned course files and completed section 2'

Because of the way I structured my code, I did not follow all the steps in the course. Below are some commands the author used that I did not use. The first one displays the remote repositories server address for fetch and push commands. The second one changes the server address for the repo. 

git remote -v
